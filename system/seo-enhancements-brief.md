# Implementation Brief — SEO Page Enhancements (v2, evolved system)

Adds eleven enhancements to the ingredient conversion pages, sequenced as **nine
independently-shippable tasks**. Hand this whole file to Claude Code. Each task can be
done, verified, and committed on its own before moving to the next.

The system has evolved since these pages were first built: there's now a **local admin
panel** that persists ingredients and **triggers a rebuild** to regenerate pages; ingredients
are split into **liquid vs. dry**; and each ingredient can carry **1–2 custom FAQs** entered
in the admin. The tasks below account for all three.

---

## STEP 0 — Inspect and map the repo first (do before any task)

Do not assume file names. Read the codebase and write down, for your own reference, where each
of these lives — every task references them:

- **Ingredient data store** (DB table / JSON / etc.) and the **ingredient record schema** — in
  particular how `type` (liquid|dry), `density`, `names`, `note`, and `customFaqs` are stored.
- **The page generator / render layer** — where a question page and a hub page get their HTML
  (titles, `<head>`, body sections, JSON-LD).
- **The shared client dataset emit** (the `data.<lang>.js` / `window.__KITCHEN_DATA__` file) and
  the **calculator + recipe-scaler** that consume it.
- **The build trigger** invoked by the admin panel, and the **sitemap** generator.
- **Any existing number formatter** (Bulgarian decimal comma) and **JSON-LD builders**.

State your mapping in the PR description so it's reviewable.

## Global conventions (apply to every task)

- **Thin-content guard.** These pages are generated by the hundreds. Only add things that
  **vary per ingredient from the data**. Anything identical on all pages must be a single short
  line, never a paragraph repeated site-wide — that triggers the thin/templated-content trap.
- **Liquid vs. dry is a first-class branch.** Respect `ingredient.type` everywhere a number is
  shown:
  - **dry** → primary reference is **grams**; volume→grams is the headline conversion.
  - **liquid** → primary reference is **milliliters**, with grams shown as a secondary value
    (e.g. "1 чаша мляко = 250 мл ≈ 258 г"). For liquids, lead with ml; for dry, lead with grams.
- **Numbers** use the project's existing **Bulgarian decimal-comma** formatter. Never print a raw
  `.` decimal in user-facing text.
- **Structured-data rules (current):** keep `BreadcrumbList` (live rich result). `FAQPage` is for
  machine/AI-crawler understanding only — **do not** expect a SERP dropdown and **do not** add
  `HowTo`. No `speakable`.
- **Keep the build green.** After each task: regenerate, confirm page count is unchanged (unless
  the task adds pages), JSON-LD still parses, and the sitemap is consistent. If `noindex`
  experiment mode is on, leave it on.
- **Admin-driven fields.** When a task needs new per-ingredient data, add the field to the
  ingredient schema **and** an input in the admin panel, so it can be filled without code.
- **Densities remain estimates** until verified — several tasks surface numbers, so wrong data
  becomes more visible. Don't remove `noindex` until densities are checked.

## TASK 1 — Methodology + "проверено на" trust line  *(E-E-A-T; fast)*

**Goal:** A one-line credibility signal the messy competitor sites lack, tied to your density work.

**Where:** Question + hub page render; ingredient schema; admin.

**Implement:**
- Add ingredient fields `source` (text/URL, optional) and `verifiedOn` (date, optional); expose both in the admin.
- Render one small line near the explainer: "Стойностите са изчислени по плътност" + (if `source`) "· източник: {source}" + (if `verifiedOn`) "· проверено на {verifiedOn}".
- Branch wording on type: dry → "по плътност (грам/мл)"; liquid → note that ml is measured directly and grams derived from density.

**Acceptance:** Line appears on every page; shows source/date only when present; no layout shift.

**Notes:** This is also your natural home for surfacing the verification pass before launch.

## TASK 2 — Site entity + page metadata schema  *(markup; fast)*

**Goal:** Make Google treat the site as one brand entity and give freshness signals.

**Where:** Homepage `<head>` (sitewide), every page `<head>` JSON-LD + render.

**Implement:**
- **Once, on the homepage:** `Organization` + `WebSite` JSON-LD (name, url, logo; optional `SearchAction`).
- **Per page:** add a `WebPage` node with `inLanguage: "bg"` and `isPartOf` pointing at the `WebSite` `@id`; include `dateModified` (from the ingredient's `verifiedOn`/last-edit timestamp) and show a small visible "обновено: {date}".
- Keep the existing `BreadcrumbList`; you may combine nodes via an `@graph`.

**Acceptance:** Homepage has Organization+WebSite; each page validates with WebPage+Breadcrumb (+FAQPage where present); `dateModified` reflects real edit time.

## TASK 3 — "Американски еквиваленти" line  *(text; free query capture)*

**Goal:** Capture "1 cup brashno в грама"-type searches and serve the translated-recipe audience — computed from existing densities.

**Where:** Question + hub page body (a small block, not a paragraph).

**Implement:**
- One compact line, computed per ingredient:
  - **dry:** "В американски рецепти: 1 cup ≈ {g} г · 1 tbsp ≈ {g} г · 1 tsp ≈ {g} г" (US cup 240 ml, tbsp 14.79 ml, tsp 4.93 ml × density).
  - **liquid:** lead with volume: "1 cup ≈ 240 мл · 1 tbsp ≈ 15 мл · 1 tsp ≈ 5 мл" and append "(≈ {g} г)" for the cup.
- Use the comma formatter; keep to one line.

**Acceptance:** Present on all pages, values match density × US unit volumes, branches correctly by type.

## TASK 4 — Computed FAQs + merge custom FAQs + units-relationship answer  *(text; integrates your custom-FAQ feature)*

**Goal:** Natural-language Q&A headings (what AI Overviews / "People also ask" pull) — combining auto-generated questions, a units-relationship answer, and the **1–2 admin custom FAQs**.

**Where:** FAQ section render + `FAQPage` JSON-LD; admin (custom FAQs already exist — just ensure they flow through).

**Implement:**
- Build the FAQ list as: **computed defaults** + **`ingredient.customFaqs`** (admin), de-duplicated, custom ones first or last consistently.
- Computed defaults (per ingredient, type-aware), e.g.:
  - dry: "Колко е половин чаша {ing}?", "Колко лъжици са 100 г {ing}?"
  - liquid: "Колко мл е една чаша {ing}?", "100 мл {ing} колко грама е?"
  - units-relationship (all): "Колко супени лъжици има в една чаша?" → "Една чаша (250 мл) = ~17 с.л. = ~50 ч.л."
- Render each as a semantic question heading (`<h2>` or `<summary>`) + one-sentence answer, and include **all** of them in the `FAQPage` JSON-LD.

**Acceptance:** Pages show computed + custom FAQs with no duplicates; JSON-LD `mainEntity` count matches the visible list; answers are type-correct and comma-formatted.

**Notes:** This supersedes the earlier two hard-coded FAQs — fold those into the computed set.

## TASK 5 — Forward ↔ reverse cross-links  *(internal linking; fast)*

**Goal:** Tie each "чаша → грамове" page to its "колко чаши са 100 г" reverse (and back), strengthening the cluster and navigation.

**Where:** Question page render.

**Implement:** On a volume→grams page, add a link to the matching reverse page for the same ingredient; on a reverse page, link back. Derive the sibling slug from the shared pair/slug helper so links never break.

**Acceptance:** Every forward page links to its reverse and vice-versa; no 404s; links survive new ingredients automatically.

## TASK 6 — Ingredient category hub layer  *(structure; medium)*

**Goal:** A second hub tier (Брашна и нишестета, Течности, Подсладители, Ядки и сухи плодове, Млечни…) for more internal-linking surface and tighter topical grouping.

**Where:** Ingredient schema + admin; new category page type; pillar + ingredient-hub render; sitemap; breadcrumbs.

**Implement:**
- Add a `category` field to the ingredient schema + an admin selector (seed the category list).
- Generate one category hub page per category at `/{lang}/merki/kategoria/{slug}/`, listing its ingredients (reuse hub card styling) with `BreadcrumbList` + `ItemList`.
- Link categories from the pillar; optionally extend breadcrumbs to Начало › Мерни единици › {Категория} › {Съставка}.
- Add category pages to the sitemap.

**Acceptance:** Each category page lists the right ingredients; pillar links all categories; sitemap includes them; adding an ingredient with a category in the admin places it correctly after rebuild.

## TASK 7 — Per-page social answer-card image  *(distribution; highest share value)*

**Goal:** An auto-generated OG/Twitter card showing the answer ("1 чаша брашно = 125 г") on your brand background — the single biggest win for Pinterest/Facebook sharing.

**Where:** `<head>` `og:image`/`twitter:image`; image generation in the build **or** an on-demand endpoint.

**Implement (pick based on stack found in Step 0):**
- **If there's a server/serverless layer** (likely, given the admin panel): an on-demand endpoint `/og/{id}/{slug}.png` that renders an SVG→PNG card from the page's data (cache aggressively). Cleanest for hundreds of pages.
- **If purely static:** generate PNGs at build time into `/assets/og/{id}-{slug}.png` (SVG template + a rasterizer), and reference them.
- Card content is type-aware (grams for dry, ml + grams for liquid). Set `og:image`, `twitter:card=summary_large_image`, width/height.

**Acceptance:** Each page exposes a unique, correct card image; it renders in a social/Pinterest debugger; build (or endpoint) handles all ingredients including newly-added ones.

**Notes:** Reuse the brand palette/fonts. This is high value but the most infra — fine to schedule after Tasks 1–6 are live.

## TASK 8 — Printable per-ingredient mini-chart + Pinterest save  *(distribution; medium)*

**Goal:** Per-ingredient printable chart (link-bait + Pinterest fuel) and a one-tap save.

**Where:** Question/hub page; print stylesheet; optional small PDF/printable view.

**Implement:**
- A "Принтирай / запази таблицата" affordance producing a clean, branded chart of that ingredient's conversions (reuse the scaler's print approach: `@media print` that strips UI to just the table + brand).
- A Pinterest "Save" button targeting the Task-7 card image so pins look good.

**Acceptance:** Print view is clean and branded; Pinterest save uses the answer-card image; works per ingredient.

## TASK 9 — "Беше ли полезно?" feedback  *(engagement; fast, do last)*

**Goal:** Low-friction quality signal + small dwell-time bump.

**Where:** End of question/hub page; a lightweight endpoint or analytics event.

**Implement:** A two-button "Беше ли полезно? 👍 👎" that records an anonymous event (analytics or a tiny admin-readable counter). No PII, no account. Optional: surface low-scoring pages in the admin so you can fix weak ones.

**Acceptance:** Voting records an event; UI gives confirmation; no personal data stored.

## Suggested order & rationale

1–2 (trust + schema) are fast, low-risk foundations. 3–5 (computed text + links) are free query
capture that lean on data you already have. 6 (categories) touches the schema/admin, so it's
mid-sequence. 7–8 (OG image, printables) are the higher-value distribution pieces but need the
most infra. 9 (feedback) is a quick finisher. Ship and verify each before the next.

**Skip list (don't implement):** `HowTo` schema, anything relying on FAQ rich snippets, `speakable`,
and repeated "about this ingredient" prose — all thin-content or dead-feature traps.
